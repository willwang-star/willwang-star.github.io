What is DDE?

🔍 Data Discovery

Search across Intuit's data catalog — tables, columns, schemas, lineage. Backed by the DDE API layer at dde-mcp-server.api.intuit.com. Think of it as Google for Intuit's internal data lake.

🔐 Data Access

Request and manage access to data assets — all from your IDE. The dde-access-mcp handles access requests to DLAP and DACM without leaving Cursor or Claude Code.

⚡ Data Exploration

Run queries and explore data interactively on Databricks. The Databricks MCP bridges your conversational AI workflow with actual compute. Exploration is backed by Databricks clusters.

How DDE fits in the Intuit stack

The data workflow today

🏢 Dev Portal — Service registry, asset management
↳ 🔍 DDE (Data Discovery & Exploration)
↳ 🗄️ Data Lake (ADAPT / Iceberg) — Raw data tables
↳ 🔐 DACM / DLAP — Data access control & management
↳ ⚡ Databricks — Interactive compute for exploration
↳ 📋 CDC (Customer Data Cloud) — User & tax attribute data

What DDE MCPs give you

  • Natural language search across the entire data catalog
  • Table schema inspection and column-level detail
  • Data lineage tracing (upstream/downstream)
  • Access request submission and status tracking
  • Databricks query execution and result analysis
  • Collection management for bounded search contexts
  • Table ownership and endorsement information
The problem DDE solves

Before DDE MCPs

😫
A data developer needed to: (1) Open DDE UI in browser, (2) Search manually, (3) Copy table names, (4) Switch to Databricks, (5) Write SQL, (6) Go back to DACM to request access, (7) Wait for approval email, (8) Switch back and run queries. Total: ~45 minutes per data question.

After DDE MCPs

🚀
Ask your IDE: "Find tables related to customer tax events." Agent searches, inspects schemas, checks access, submits request if needed, and drafts a query — all in one conversation. Total: ~4 minutes.

DDE MCP Tools

Four MCP servers cover the full data workflow: authenticate, discover what exists, request access, and run queries. Install once, use everywhere your AI agent runs.

💡
DAST Plugin available: All four MCPs are bundled in the intuit-data-discovery-access-exploration-capability-plugin ↗ — the easiest way to get all DDE capabilities at once. Also available via MCP Marketplace ↗.
MCP 1 — Data Discovery
data-discovery-mcp
Search Intuit's entire data catalog using natural language. Find tables by business concept, inspect schemas, explore column metadata and data lineage — all without leaving your IDE.
Primary MCP 14 tools 80%+ accuracy
codegen mcp install data-discovery-mcp:latest
Server endpoint: dde-mcp-server.api.intuit.com · Proxy: localhost:8098

Available tools

searchTables getTableSchema getColumnDetails getDataLineage searchCollections getTableOwner getAssetMetadata searchWithinCollection
⚠️
Known limitation: searchTables has a hard cap of 100 results with no pagination. No filterByOwner support yet. Work in progress per team.
MCP 2 — Data Access
dde-access-mcp
Request, check, and manage access to data tables — directly from your IDE. Interfaces with DACM (Data Access Control Management) and DLAP. Requires Dev Portal service onboarding and a separate INTUIT_API_KEY.
Access Control Service onboarding required
# Dev Portal onboarding required first # Ask in #dde-agentic-discovery-exploration-users for API key codegen mcp install dde-access-mcp:latest

Available tools

request_access check_access_status list_my_access get_table_owner cancel_access_request
💬
Feature request (from users): request_access currently doesn't return the approver's name. You have to go to the DDE UI to find the owner. Community has requested this be surfaced in the API response.
MCP 3 — Databricks Exploration
databricks-mcp
Run SQL queries, analyze data, and interact with Databricks clusters from your IDE. Pairs with data-discovery-mcp for a complete discover → access → explore workflow. Best used after you have table access confirmed.
Exploration UC adoption required
⚠️
Databricks Connect is blocked until Unity Catalog (UC) Adoption launch. For full interactive compute, local laptop connections are currently the interim path. Follow #dde-agentic-discovery-exploration-users for updates.

Available tools

execute_query list_catalogs list_schemas describe_table get_query_history create_table
🏆
GED success story: The Agentic Data Discovery & Exploration GED project demonstrated using Databricks MCP + data-discovery-mcp together for end-to-end data workflows. See the demo in GED project dashboard ↗.
MCP 4 — Authentication
auth-mcp
Handles authentication and token management for DDE services. Required as a companion server alongside data-discovery-mcp and dde-access-mcp to keep your IAM session alive and refresh tokens automatically without leaving your IDE.
Auth Required companion
codegen mcp install auth-mcp:latest
Install alongside the other DDE MCPs for seamless auth management

Why you need it

🔑
Without auth-mcp, your session tokens can expire mid-workflow, causing silent failures or AuthenticationFailed errors. This server refreshes credentials in the background so long-running discovery sessions don't break. See the full setup guide at Confluence MEDO/3590526266 ↗.
⚠️
HCAP environments: Exercise extra caution using auth-mcp in high-compliance contexts. Review the latest guidance in #dde-agentic-discovery-exploration-users before use.

Getting Started

Step-by-step setup for data-discovery-mcp — the most commonly used DDE server. Full guide: github.intuit.com/dataundstd-dmsearch/dde-mcp-guide ↗

💡
Fastest path: Install the DAST plugin — it bundles data-discovery-mcp, dde-access-mcp and all dependencies in one shot. Enable once, works in Cursor and Claude Code (CLI).
🍎
Apple Silicon (M1/M2/M3) users: If you have an Intel-era Homebrew installation at /usr/local/bin/brew, you must uninstall it and reinstall the ARM version at /opt/homebrew before proceeding. Mixing architectures causes silent failures during Podman and codegen setup. Request temporary admin via ServiceNow before starting. Full guide: Confluence MEDO/3590526266 ↗
Setup steps — data-discovery-mcp
1
Prerequisites — eiam, codegen, node, yarn
You need the Intuit eiamcli and codegen CLI (v0.74.3+), plus node and yarn. Check versions first. Install from Dev Portal if missing.
eiamcli --version # Intuit IAM CLI codegen --version # should be 0.74.3 or newer node --version # required for codegen internals yarn --version # required for some MCP installs
2
Log in with eiamcli
IAM authentication is required before the proxy can communicate with DDE services. Refresh your ticket if you see 401 errors later.
eiamcli login # first-time login eiamcli iamticket # refresh ticket when it expires
3
Install and start Podman
DDE MCPs require a container runtime. Podman is the recommended option (Docker also works). You must initialise and start the Podman machine before running codegen installs — otherwise the setup fails silently.
# Install Podman Desktop from https://podman-desktop.io or via brew brew install podman podman machine init # initialise VM (one-time; may fail on bad gateway — retry) podman machine start # start the VM before codegen installs podman info # verify it's running
⚠️
Podman machine restart clears containers. If you restart your laptop and the MCP stops working, re-run podman machine start and then toggle the MCP off/on in your IDE.
4
Install all four DDE MCPs via codegen
Install each server in order. auth-mcp handles token refresh and is required alongside the others. intuit-remote-mcp-proxy is installed automatically with data-discovery-mcp. When prompted, select your IDE (Cursor or Claude Code).
codegen mcp install auth-mcp:latest codegen mcp install data-discovery-mcp:latest codegen mcp install dde-access-mcp:latest codegen mcp install databricks-mcp:latest # Select IDE when prompted for each # If NOT prompted for API key, see troubleshooting below
5
Configure your INTUIT_API_KEY
The codegen installer sometimes skips the API key prompt. If your searches fail with "Failed to search data entities", manually set the key. Request one in #dde-agentic-discovery-exploration-users — it starts with prdaky....
# Check your mcp.json — it should look like this: "data-discovery-mcp": { "url": "http://localhost:8098/api/mcp" }, "intuit-remote-mcp-proxy": { "command": "codegen", "env": { "INTUIT_API_KEY": "prdaky..." }, "args": ["mcp", "remote-init"] }
🚨
Common mistake: The URL should be http://localhost:8098/api/mcp — NOT http://localhost:8098/data-discovery-mcp. The proxy must not have the MCP name appended to the path. Claude Code users especially watch out for this routing issue.
6
Add Cursor Rules for DDE
The DDE team maintains a Cursor Rules file that gives your AI agent best-practice guidance on how to use the DDE MCPs correctly — including which tools to call, how to handle IRNs, and when to pause for access approval. Add it to your Cursor project.
# In Cursor: press cmd+shift+p → "Add Cursor Rule" # Paste the rule from GitHub: # github.intuit.com/cognition-discovery/discovery-docs # → dde-data-discovery-rules.md
💡
The rules file significantly improves agent behavior — it teaches Cursor how to chain discovery → access → query workflows without hallucinating tool names or IRN formats.
7
Start the remote proxy and verify
Enable/toggle the MCPs in your IDE. The remote proxy should start on port 8098. Verify all 14 tools from data-discovery-mcp load successfully.
# Proxy runs on localhost:8098 # In Cursor: Settings → MCP → toggle data-discovery-mcp to green ✅ # In Claude Code CLI: tools should auto-load on session start
8
Test with a sample query
Try asking your agent to find a table. If you see results, you're good. If not, check the troubleshooting section.
# Example prompts to test your setup: "Find tables related to customer tax filing events" "What columns does intuit_foundation_identityandcustomer360... contain?" "Search for tables in the adapt_7216 schema"
Using DDE MCPs in Claude Code (CLI)

Claude Code CLI uses a slightly different MCP configuration than Cursor. The claude mcp add --scope user command registers MCPs globally for your user. Use these commands after completing the codegen installs above.

A
Register MCPs with Claude Code CLI
# Register the remote proxy as a user-scoped MCP claude mcp add --scope user intuit-remote-mcp-proxy \ codegen mcp remote-init # Register data-discovery-mcp pointing to the proxy claude mcp add --scope user data-discovery-mcp \ --url http://localhost:8098/api/mcp # Register auth-mcp claude mcp add --scope user auth-mcp \ codegen mcp auth-init # Verify all MCPs are registered claude mcp list
B
Copy Cursor-configured MCPs to Claude Code
If you already have MCPs set up in Cursor, there's a faster path — copy the MCP config directly. See the Confluence guide for details on syncing configs between IDEs.
# Full guide: wiki.cloud.intuit.com/wiki/spaces/MEDO/pages/3591559561 # "Adding Cursor Configured MCP Servers to Claude Code CLI"
📌
Claude Code CLI only supports Streamable HTTP transport for DDE MCPs — this works. The Claude Desktop app does not support this transport. Do not confuse Claude Code CLI (terminal) with Claude Desktop (GUI).
Compatible IDEs

✅ Cursor

Fully supported. All 14 tools work. The primary recommended IDE for DDE MCP usage. Best experience for interactive data discovery workflows.

✅ Claude Code (CLI)

Fully supported. Works with the remote proxy on localhost:8098. Watch out for the URL routing issue (see step 4). Most users on Claude Code report it working after fixing the path.

❌ Claude Desktop App

Not yet supported. Claude Desktop does not support the Streamable HTTP transport that DDE MCPs use. Use Claude Code CLI as the alternative.

Common use cases

🎯 New feature research

When starting a new feature, ask your agent to find all tables related to a business concept. Get schema details and understand data shape before writing any code.

"Find all tables related to customer tax filing events. Show me the top 5 most relevant with their schemas."

🔗 Data lineage tracing

Understand upstream/downstream dependencies before making schema changes or building pipelines. Avoid breaking downstream consumers accidentally.

"Show me the lineage for adapt_7216.member_events. What tables depend on it?"

🔐 Access workflow (agentic)

Ask your agent to check if you have access, identify the owner, and submit a request — without switching tools. Track status in the same conversation.

"Check if I have SELECT access to member_profile. If not, request it and tell me who the approver is."

📊 Team ownership audit

Find all tables owned by a specific team or project for maintenance or migration planning. Useful for leads doing data stewardship.

"Find all tables owned by the identity360 team. List them with their current access status."

🔍 Column-level search

Search for specific column names or data types across the entire catalog. Useful when you know what attribute you need but not which table has it.

"Which tables have a column called 'tax_year' or 'filing_status'? Show schemas."

📦 Collection-based search

Use Collections to create a bounded search context. Search within a pre-defined set of trusted tables rather than the full catalog. Great for team-specific workflows.

"Search within my 'tax-platform' collection for tables related to deductions."

Access workflow gotchas

The access lifecycle has a known pain point that many users hit. Here's the recommended approach while the team works on full agentic support:

The problem

  1. Agent runs a query — gets INSUFFICIENT_PERMISSIONS
  2. Agent calls request_access — gets APPROVAL_REQUESTED
  3. Access approval takes 3–4 days
  4. User forgets which query they needed, or forgets to follow up
  5. No way to track pending requests agentically

Recommended workaround

  • Save the full query in a file when access is requested
  • Note the table IRN and requested time
  • Use check_access_status periodically to poll
  • Go to DDE UI to find the approver's name and follow up directly
  • Consider using list_my_access at session start to know what's available
💬
The team is tracking a feature request to surface approver info in the request_access response. Vote for it and follow progress in #dde-agentic-discovery-exploration-users.

Key findings — Hey Marvin research (Project 35183)
📊
Research scope: DDE user interviews and follow-me-home sessions analyzed in Hey Marvin Project 35183 — "Data Platform Research." Findings represent real user workflows, friction points, and impact measurements collected across Intuit data teams.
🧠 Theme 1 — Tribal knowledge is the default discovery mechanism
Without DDE, analysts depend heavily on institutional knowledge held by a small number of senior colleagues. New joiners have no systematic way to find relevant data outside their immediate team's domain. This creates asymmetric productivity and slows cross-team data reuse.
⛓️ Theme 2 — Multi-team access requests create cascading delays
Getting set up for a single use case often requires reaching out to three or four different teams for access to different data assets. Each handoff adds waiting time and context-switching. Users lose the thread of what they were originally building.
⏱️ Theme 3 — Access approval takes up to a week for new joiners
Manual approval workflows require human sign-off and can take up to one week for basic table access. This delay directly blocks new hires from being productive during their critical onboarding window. DDE's agentic workflows aim to collapse this to minutes for approved schemas.
🚀 Theme 4 — DDE MCPs dramatically reduce time-to-insight
Users who successfully onboard to DDE MCPs report the biggest single quality-of-life improvement in their workflow. The shift from manual browser-based discovery to natural language search collapses a 45-minute multi-tool workflow to under 5 minutes. This is the core value proposition confirmed by user interviews.
📋 Theme 5 — Onboarding delay compounds the productivity gap
New team members feel the impact most acutely. When the tools that would accelerate onboarding are themselves difficult to onboard to, the productivity gap widens. Users interviewed noted that having DDE from day one would have significantly shortened their ramp-up time.
🔍 Theme 6 — Natural language search removes the "knowing the right words" barrier
Historically, finding data required knowing the exact schema name, team ownership, or Intuit-internal naming conventions. DDE's natural language search means analysts can describe what they're looking for in business terms, not technical identifiers. This opens the catalog to a much wider audience.
🔐 Theme 7 — Access approval interrupts agentic workflows
Even after setup, the access request → wait → resume cycle breaks the agentic flow. There is no in-IDE way to track pending approvals, and the approver's identity isn't surfaced. Users manually save intermediate state to files and come back days later — if they remember where they were.
✅ Theme 8 — Trust signals matter as much as discoverability
Finding a table is only half the job. Users need confidence that the data is accurate, endorsed, and actively maintained. Endorsement flags and AI-generated table descriptions partially address this, but sparse or stale metadata erodes trust in the MCP's results — even when the search itself is correct.
