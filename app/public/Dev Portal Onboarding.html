5 consequences of fragmentation — downstream impact

From the Platform Vision doc (Rishav Mittal). Downstream systemic effects — each maps to a design principle you'll be solving for.

1 · Inconsistent UX
Different design patterns, auth flows, and IA across each portal. IDS tokens not universally adopted — compounding cognitive load with every cross-tool workflow.
2 · Duplication of Effort
Search, login, auditing, and notifications are independently rebuilt in each portal. Maintenance overhead compounds at scale as each vertical evolves separately.
3 · Hindrance to Cross-Domain Innovation
Combining AI Workbench + DDE + IXP in a single workflow has no native integration path. Everything is bespoke, slowing the pace and scope of innovation.
4 · Inefficient Discovery
No unified search spanning all portals. Developers hunt for APIs, datasets, and tooling across separate systems — "The Treasure Hunt" is step #1 in almost every persona journey.
5 · No Single Pane of Glass
No consolidated view of projects, metrics, or compliance. Developers toggle between 10+ interfaces for a complete picture. Accountability is fragmented — no single ownership view.
8 core systemic problems — fresh-eyes checklist
Context Switching Tax
Moving DevPortal → AIW → DDE → IXP → Databricks for a single workflow. Each tool has its own mental model. Average workflow spans 3–5 tools.
Discovery Friction
Docs scattered across Confluence, Slack, Autura, DevPortal. "The Treasure Hunt" is step #1 in almost every JTBD workflow across all personas.
Access & Approval Latency
FY23 SLA: 5-day wait for data access. Manual credential/permission flows via DACM stall momentum for every new project.
Fragmented Access Management
Each portal manages its own permissions independently. No unified layer for granting, auditing, or revoking access — creating compliance risk and operational overhead.
UX Inconsistency
Same pages (homepage, search, create) built differently across tools. IDS tokens not adopted everywhere. 10+ component unification opportunities found in audit.
No Unified Monitoring
10+ dashboards to answer "Are we compliant?" Splunk, Wavefront, Data Pulse, ServiceNow all siloed. 1.5 hrs avg per P1 investigation.
No Customer Impact Visibility
Builders can't trace a microservice → customer-facing feature → business outcome. "Show the why behind the what" is aspirational, not yet real.
Duplicate Effort on Shared Concerns
Each portal independently builds search, login, auditing, and notifications. A unified platform should solve these once and share across all tools.

Technical foundation: the federated shell architecture

The Platform Vision doc (Bhavana Reddy Gajjala, Rishav Mittal) defines how the IA you're designing is technically realized. DevPortal transitions from a "loose federation of links" into a Federated Container Architecture. The Shell is the OS for the engineering lifecycle. Verticals (AI Workbench, IXP, Data Discovery, etc.) evolve from standalone websites into Workbench Modules running inside the governed shell.

Shell responsibilities (horizontal layer)
Global Navigation & Header
The Shell owns the top nav, global search, Intuit Assist, notifications, announcements, and profile. These render for every persona, on every vertical, consistently. The Shell team (not workbench teams) governs this surface.
Context SDK — Shared Global State
The Shell exposes a Context SDK that provides every hosted workbench with unified identity (logged-in user + RBAC), delivery team context, and user preferences. Changing your team in the global header propagates to all loaded workbenches — this is what makes "contextual left nav" technically possible.
PDLC Tracker & Intent-Based Navigation
A persistent Shell component that maps whatever tool the user is currently inside to a lifecycle stage (Plan → Code → Build → Deploy → Monitor). Navigation is intent-centric, not tool-centric. This component lives in the Shell, not in the workbenches.
Workbench integration patterns (vertical teams)
Pattern A — Core Widget (top-down)
Each vertical exposes its entire app as one root widget. DevPortal core loads it via WidgetLoader. AI Workbench POC demonstrated this works — the vertical team owns state, routing, analytics, and logging entirely within their widget. DevPortal just mounts it. Preferred approach.
Pattern B — Module Federation
Each vertical exposes a remoteEntry.js bundle via Webpack Module Federation. The Shell dynamically loads these without requiring plugin dependencies in the web app. Successfully tested. Preferred for teams that can't refactor to a single core widget.
Domain Redirection Strategy
Users with bookmarks to ai-workbench.app.intuit.com or experimentation.app.intuit.com will be redirected to devportal.intuit.com/app/dp/wb. Two approaches: Gateway Redirection (zero vertical team effort, preferred) or Software-Level Redirection. NetGenie handles gateway redirects.
💡
Design implication: Every navigation decision you make has a Shell vs. Workbench ownership question behind it. "Who governs this surface — the shell team or the vertical team?" is a recurring design principle question. The answer determines whether a pattern is standardized once (Shell) or can vary by workbench (Vertical). When in doubt, Shell wins.

Guiding Principles — The Design Mindset

The team's explicit agreed principles. Know them cold — they'll shape every design decision and every stakeholder conversation. Each principle includes a concrete Dev Portal application below.

🧠 Mindset
One Experience, Not Just One URL
"Design the experience, not the directory."
True unification is not about putting all tools under one domain. It's about making the boundaries between tools feel invisible. A unified URL was Bronze. Unified experience is Gold.
In DevPortal: A developer starting a project in Build should be able to transition to AI Workbench's Playground and back to DevPortal's Manage view without ever feeling they changed apps. Navigation state, context, and terminology must be consistent across all tool surfaces.
Anticipate Needs, Don't Just Present Tools
"Think ahead for the builder."
Move from a reactive catalog to a proactive, intelligent assistant. The "Next Best Step" Intent Framework proactively suggests the next logical action based on context, role, and project goals. Foundational for future AI agents performing multi-step tasks on behalf of the user.
In DevPortal: When a developer registers a new AI Agent, the platform should proactively surface: "You'll also need to request PrivateAuth+ access, configure circuit breakers, and run an ADAPT sandbox test" — not wait for them to discover this by failing three times.
Connect Every Action to Customer Value
"Show the 'why' behind the 'what.'"
Every persona should be able to trace a microservice/library → customer-facing feature → business outcome. IA should be organized around the customer, not internal technical assets.
In DevPortal: The Teams → Features (PDLC Tracker) view must show: which microservices power which product features, which customer segments they affect, and what the measured impact is. This turns DevPortal from an engineering tool into a business tool.
Empower the Community to Build the Platform
"Built by many, guided by one."
DevPortal should not be top-down or centrally controlled. It should be an open platform that harnesses collective developer expertise — contribute, share, and reuse over reinvent.
In DevPortal: Security, Observability, and Paved Roads teams each own their feature areas within DevPortal. Your role as Principal Designer is not to design every screen — it's to govern the shared design language, set the contribution model, and ensure consistency across all team contributions.
⚙️ Execution
Adaptive to Every Builder
"One system, many expressions."
Design flexible, persona-aware experiences that adjust to context, role, and need — without fragmenting the ecosystem. Same underlying IA, expressed differently per persona. Do not create separate portals; create adaptive surfaces.
In DevPortal: Home must render a completely different experience for a Data Engineer vs. a new SWE vs. an Engineering Manager — all from the same URL and same underlying IA. This is solved at the persona detection layer, not by building separate apps.
Cohesive by Design
"Consistency in patterns, continuity in journeys."
Create seamless, end-to-end flows that make multiple tools feel like one connected experience. Cross-tool transitions should feel like navigating a feature, not switching an app.
In DevPortal: The Silver phase audit found 20+ pages and 10+ component patterns inconsistently implemented across DevPortal, AI Workbench, DDE, IDX Studio, and IXP. Srini Chakkarapani's IDS standardization workstream is the execution of this principle.
Intelligent by Nature
"Smart guidance, not noise."
Infuse AI thoughtfully to anticipate intent, reduce friction, and surface meaningful insights. Intuit Assist is ambient — embedded in context throughout, not just a chat box in the corner.
In DevPortal: Intuit Assist should be present at every friction point: helping a developer find the right API in Products, explaining a compliance gap inline in Monitor, suggesting the next action in Build based on project context. It's an ambient layer woven into each surface, never a separate destination.
Simplify Relentlessly
"Design for focus, not feature overload."
Show what matters — no more, no less. The audit found 20+ pages of inconsistency across tools. This job is as much about subtraction as addition.
In DevPortal: The Silver phase is not just about adding new surfaces — it's about removing redundant ones. Every duplicate experience found in the audit (duplicate search bars, duplicate Create flows, duplicate dashboards across AIW and DevPortal) is a subtraction opportunity. "What can we remove?" is as important as "What should we build?"
The LEGO analogy — the team's core mental model
The Consumer: Shopping for LEGO Parts
A builder who needs a specific pre-built part to finish their creation. They aren't looking to invent a new brick — they want to find an existing one that solves their problem right now.

Job: Find a high-quality, pre-built technical artifact they can trust and easily snap into their project. Avoid reinventing the wheel.
The Creator: Building with Raw LEGO Bricks
The builder has a brand new idea. They aren't looking for a pre-made part — they are starting with raw materials to build a new component that has never existed before.

Job: Build a new, high-quality, reusable technical artifact from the ground up that solves a new problem and makes it available for others.
💡
The portal must serve both mindsets simultaneously and often within the same session. Never assume which mode a user is in when they land on any page. Persona-based Home + intent-based navigation solves for this.
"We're not unifying tools. We're unifying intent."
Users should do work, not choose tools. Stop creating "hubs" as destinations and replace them with intent, context, and persona gravity.
Contributing platform model: DevPortal is contribution-based. Security, Observability, Paved Roads, and other platform teams build their own features within the portal. Your role as Principal Designer is governing the cohesive experience across all contributions. | IDS adoption: Not all tools have adopted IDS tokens and standards yet.
