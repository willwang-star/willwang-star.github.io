AI Workbench (AIW) — Onboarding

Intuit's internal developer platform for AI and ML development. Lives inside Dev Portal. Your entry point for building, testing, and shipping AI agents at Intuit. This brief covers everything — from current state to where the platform is heading and who's driving it.

What is AI Workbench?
🤖 GenAI / Agent Development

The primary development surface for building LLM-powered agents. Model Discovery, Skills Registry, Prompt Registry, and Langfuse eval integration all live here. Evolving toward local-IDE-first workflows.

🔬 MDLC — Classical ML

Machine Learning Development Life Cycle tooling. Experiments, AutoML, Endpoints, Batch Inference, Model Registry, and Pipelines. Currently project-scoped; migrating to asset-scoped routing (see Roadmap).

📊 Model Discovery

Live leaderboard for LLM comparison and selection. Browse available models, compare performance, and understand which LXS models are available for your use case. Currently sparsely populated but the foundational piece.

Where AIW fits in the Intuit stack

Part of Dev Portal (DevP)

AI Workbench lives inside devportal.intuit.com as a sub-platform. AIW is the specialized workspace for AI/ML work within that ecosystem.

The AI Development Design Vision

Sooji Son & Elmer KimNii · Dec 5, 2025 · Full doc ↗

⭐
The core proposition — Don't build a monolithic "AI IDE." Instead, build a guided, AI-assisted developer experience that connects existing tools, documentation, and workflows into a coherent step-by-step journey. Orchestrate and guide, don't replace.
Why NOT "just build an AI IDE"?

The argument against a single IDE

  • Each platform (Databricks, SageMaker, Langfuse) is deeply specialized — no IDE can match that depth
  • Developers already have a working ecosystem. These are already their dev environment.
  • What users need is a cohesive experience across environments, not a new environment that replaces them

What to build instead

  • Step-by-step onboarding flows connecting existing tools and APIs
  • An AI assistant integrated with Intuit's docs corpus to answer "how do I…" with Intuit-specific context
  • Contextual guidance that adapts based on the user's current task or tool
  • Example workflows that auto-generate starter notebooks and API calls
  • A lightweight desktop agent providing contextual suggestions (long-term)
The 3 big ideas
1 — Query Knowledge

Leverage agents to search, chat, and start tasks & workflows. Ask "how do I…" and get an answer with Intuit-specific context.

2 — Learn Through AI Hub

A structured knowledge repository with models, patterns, guides, and examples. The single source of truth for AI development at Intuit.

3 — Workflow-Centered Design

Center the experience on workflows and tasks, not on assets or features. Reduce context-switching and cognitive overhead.

The 4-stage architecture: HOME → LEARN → EXPERIMENT → MEASURE

🏠 HOME

Example projects & agents. Browse guided experiences by job-to-be-done. Quick-start paths. Browse models, prompts, RAG knowledge bases.

📚 LEARN

For any guided experience: get a preview, video, and execution plan. Structured onboarding paths per JTBD. Searchable, AI-friendly documentation.

🧪 EXPERIMENT

Automatic setup — no manual copy-paste. Preconfigured env, service tokens, variables, Databricks integration. RAI guardrails built in.

📊 MEASURE

Evaluation (Langfuse integration). Create datasets, performance dashboards. Proactive recommendations: "Your agent is hallucinating on X."

Design principles (6)
🚫 Do Not Migrate
Don't force users onto a new platform. Build bridges to where they already are.
🛣️ Opinionated With Options
Default to guided best practices, paved paths, and legal/security guardrails. Allow advanced options on demand.
✨ Elegant Yet Functional
Present the most important information first. Provide sources and details to build confidence — only when needed.
⌨️ Type > Click
Prefer keyboard-driven interaction (tab, type, hotkeys, fuzzy search) over endless dropdowns and modals.
👥 Single & Multiplayer
Empower individuals as end-to-end builders. Also provide options to invite others for legal review and output evaluation.
🔄 Forgiving
Experimentation is cheap, reversible, and safe. Preview changes, revert to previous states easily.

The 5 user segments

⚡ Power Users (N=77)

High breadth AND high frequency. ≥5 core features + ≥5 sessions in 30 days. AIW's most valuable users — and even they report friction.

🔁 Veteran Users (N=275)

Meaningful, repeated usage. ≥3 features + ≥3 sessions. They work around the friction but are at risk of sliding into lapsed.

💥 Burst Users (N=405)

Sampled heavily but didn't return. ≥3 features but ≤1 session. Tried AIW once, couldn't find value fast enough, moved on.

💤 Lapsed Users (N=717)

Previously active, haven't come back. The largest segment. Many just "forgot about it."

👀 Should-Use-But-Don't

Developers/technical roles with zero recent AIW usage. The untapped audience — would benefit from AIW but have never engaged. Key target for the Day 0 plugin.

💡
Key finding: Non-adoption is NOT driven by non-technical users. The majority of burst and lapsed users are hands-on engineers. The challenges are value clarity and getting-started friction.
Why users dropped off — 3 categories
❶ Unclear value proposition — #1 driver (10 mentions)
"Product didn't meet my needs / missing features / didn't see value" — all describe the same underlying sentiment: the value proposition was not clear or strong enough, fast enough. This is NOT a feature gap — it's a communication and discoverability problem.
❷ Hard to understand / get started — #2 driver (4 mentions, burst users)
Users dropped off before reaching value because the learning curve was too steep. Without an easy way to learn, external tools with better docs became more attractive. "Most internal tools are hard to adopt because we can't use external AI to help figure out how to use them."
❸ No longer needed / workflow changed — #3 driver (7 mentions, lapsed)
AIW isn't sticky enough to survive role or workflow changes. Also 2 mentions of "I forgot about it" — AIW isn't embedded in any core workflow and lacks re-entry hooks.
What users are actually saying
"What do you want me to use it for? I don't understand outside of setting up prompts. Is there something specific you want it to be used for regularly?"— Burst/Lapsed user
"Most internal tools are hard to adopt because we can't use external AI to help us figure out how to use them. The tools change constantly, which is fine, but the wiki pages never keep up. While I can learn any external library or tool quickly, I don't have that option for internal stuff."— Burst user (Vladimir Vainer)
"If I cannot use AI to learn how to use it, most probably I'll try to find another tool outside of Intuit's solution."— Burst user
"It is REALLY difficult to create a RAG database and to get AI agents into production. Current documentation has cost our teams weeks of wasted time."— Power/Veteran user
"Make it easier to understand and onboard for non-coders. Analysts have a need and desire to leverage LLM tools but we are frankly confused on where and how to start. If there is a 'paved path' it seems to only be paved for engineers."— Power/Veteran user (analyst)
"Had some struggle when I started using it but now I am able to figure out pretty much everything by myself since I have got some practice on it."— Power user (tribal knowledge workaround)
Key research conclusions
⚠️
The learning curve is too high for a self-serve tool. Users are NOT asking for more features, NOT primarily complaining about bugs. They couldn't quickly understand why this tool mattered for their work, and it took too much effort to figure out on their own.
✓
There is no huge feature gap — it's a discoverability and clarity problem. "The problem we hear is that it isn't a feature gap. It's a problem with discoverability and clarity of purpose." The fix is better onboarding and clearer value communication, not more features.

Research interview principles (for future studies)

Don't ask:

  • "Why aren't users using Feature X?"
  • "Why is the number dropping?"
  • "Why aren't you using our product?"

Ask instead:

  • "How are you currently accomplishing the jobs to be done?"
  • "What tools and workflows are you actually using day to day?"
  • "Walk me through how you currently get [job] done."

Agent Development for the Day 0 Experience (Plugin-First)
● POC Complete Melani Armstrong + Sooji Son · Apr 14, 2026 Full PRD ↗ 🎬 Target state demo ↗
💡
The approach: Rather than building a new surface, we bring the agent development workflow directly into AI coding tools (Claude / Cursor). A developer says "I want to build an agent today," and Claude orchestrates the entire setup on their behalf using Intuit's existing infrastructure.

Why plugin-first

Zero new surfaces

Builders work in Claude via their terminal. No new UI to learn or maintain.

Meet builders where they are

Already in Claude or Cursor. Plugin works with whichever they prefer.

Compounds over time

Every improvement to the AI model makes the plugin more powerful for free.

LLM & agent agnostic

Orchestrates setup but doesn't constrain model or framework. Any LXS model works.

The 8 builder requirements

  1. Trigger agent setup from AI coding assistant — intent detected via natural language, prerequisites installed automatically, no manual navigation to DevPortal or AIW required.
  2. Create or connect an AI experience without leaving the coding environment — DevPortal registration, LXS onboarding, Langfuse onboarding, and IDPS secrets all configured automatically.
  3. Discover and install skills using natural language — search Skills Registry by description or Skill ID. Installed and wired in automatically.
  4. Add a prompt by generating one or loading from the registry — Option A: describe agent's goal → assistant generates, publishes as v1.0, configures. Option B: provide Prompt ID → loads via Prompts SDK. Both registered and versioned from day one.
  5. Run and test locally without manual auth or Postman setup — runs with a single command. IDPS, prompt loading, and LXS connectivity confirmed by assistant. All runs logged to Langfuse.
  6. Run a quick eval from Langfuse traces during active development — select time window and metrics; assistant pulls traces, runs lightweight tests, surfaces results in context.
  7. Run a full eval against a golden dataset — provide a Google Sheet link; assistant pulls dataset, runs all tests, logs to Langfuse. Exportable for domain expert review.
  8. Commit work and open a PR from within the assistant — assistant stages all files, pushes to Git, opens a PR. No manual Git commands required.

5 plugin design principles

1. Plugin-first, not a new surface
The workflow lives inside existing AI coding environments. Not building a new UI to compete with tools builders already trust.
2. LLM & agent agnostic
Orchestrates setup but doesn't constrain the agent's model or framework.
3. Zero manual infrastructure steps
If a builder has to leave their coding environment to complete setup, that is a bug, not a workflow.
4. Every artifact registered from day one
Prompts, skills, and agents are tracked in their respective registries from the moment they're created.
5. Guide, don't gate
The assistant surfaces next steps and suggestions but doesn't force a rigid order.
